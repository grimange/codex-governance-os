# Layer 5 Codex Collaboration Operating Model

## Purpose

This doctrine defines the canonical Layer 5 collaboration operating model for
Codex work inside `codex-governance-os`.

Layer 5 establishes how the Layer 4 roles collaborate during governed
repository execution. It does not establish runtime orchestration, autonomous
delegation, or a multi-agent execution engine. It defines the bounded
collaboration rules that future Codex work must preserve.

## Scope

This canon governs:

- canonical collaboration workflow between Codex roles
- role handoff discipline
- authority preservation across collaboration steps
- restriction propagation across role transitions
- evidence preservation across collaboration outputs
- fail-closed escalation behavior during collaboration

This canon does not:

- prove runtime multi-agent orchestration
- require automatic role delegation
- replace Layer 0 through Layer 4 governance boundaries

## Layer Position

Layer 5 sits above:

- Layer 0 doctrine and safety canons
- Layer 1 interpretation canon
- Layer 2 execution surfaces
- Layer 3 Codex rules canon
- Layer 4 Codex role model

All Layer 5 collaboration behavior remains subordinate to those lower layers.

## Collaboration Model Overview

The governance OS adopts a structured Codex collaboration model.

Roles collaborate as bounded contributors within a governed execution workflow.
Specialization improves clarity, but specialization does not create new
authority. Collaboration must preserve lower-layer restrictions, evidence
boundaries, and governed routing at every step.

The canonical collaboration flow is:

- planner defines the governed execution path
- integrator aligns canonical surfaces
- verifier validates structural and evidence sufficiency
- reporter records bounded evidence and verdict outputs
- restriction keeper preserves fail-closed boundaries throughout the workflow
- architecture auditor protects doctrine integrity when architecture-sensitive
  work is involved

Roles must not bypass governed pipelines in order to accelerate work.

## Canonical Role Collaboration Workflow

### Governance Planner

Responsibilities:

- interpret instructions
- determine governance relevance
- select the next valid governed path
- preserve execution sequence across related work

Planner outputs:

- recommended pipeline execution path

Planner limits:

- may not declare verification verdicts
- may not mutate architecture doctrine outside the governed task
- may not bypass governed routing

### Surface Integrator

Responsibilities:

- update canonical discoverability surfaces
- align governance doctrine entry points
- keep canonical references synchronized

Integrator outputs:

- aligned surface updates across:
  - `README.md`
  - `docs/governance/architecture-doctrine.md`
  - `.codex/AGENTS.md`

Integrator limits:

- may not broaden claims while integrating surfaces
- may not erase explicit restrictions or non-claims

### Governance Verifier

Responsibilities:

- inspect pipeline artifact bundles
- confirm evidence presence and structural sufficiency
- validate that restrictions remain visible

Verifier outputs:

- verification verdict recommendation

Verifier limits:

- may not author structural establishment doctrine by itself
- may not reshape evidence to fit a preferred verdict

### Evidence Reporter

Responsibilities:

- record evidence artifacts
- author final bounded verdict outputs
- produce inspectable execution summaries

Reporter outputs:

- completed artifact bundles with explicit verdicts and restrictions

Reporter limits:

- may not broaden verified scope
- may not remove lower-layer restrictions from final reporting

### Restriction Keeper

Responsibilities:

- detect unsupported, ambiguous, or stale-sensitive work
- enforce fail-closed behavior
- preserve lower-layer restrictions throughout collaboration

Restriction Keeper outputs:

- restriction notices
- explicit fail-closed boundaries
- safe escalation recommendations

Restriction Keeper limits:

- may not override supported execution without evidence
- may not invent restrictions beyond the governing canons

### Architecture Auditor

Responsibilities:

- ensure architecture-doctrine preservation
- detect structural mutation attempts
- guard architecture authority during collaboration

Architecture Auditor outputs:

- doctrine alignment confirmation
- architecture-boundary warnings when needed

Architecture Auditor limits:

- may not introduce new architecture authority unilaterally
- may not broaden doctrine through inference alone

## Role Handoff Protocol

Codex roles must follow a deterministic handoff protocol:

1. planner selects the governed path
2. integrator aligns canonical surfaces required by the lane
3. verifier inspects outputs and evidence sufficiency
4. reporter records verdict and execution evidence

Restriction keeper and architecture auditor operate as continuous oversight
roles across the workflow rather than isolated terminal steps.

Roles must not bypass the defined collaboration sequence when the sequence is
relevant to the governed task.

## Authority Preservation Rules

During collaboration:

- lower-layer governance rules always prevail
- role specialization must not create new authority
- structural repository mutation must occur only through governed paths
- no collaboration handoff may weaken an existing fail-closed boundary

Layer precedence remains:

Layer 0 > Layer 1 > Layer 2 > Layer 3 > Layer 4 > Layer 5

## Restriction Propagation Rules

If a restriction is identified by any role:

- it must propagate across the collaboration workflow
- downstream roles must respect it
- the final verdict must preserve it explicitly
- later summaries must not silently narrow or erase it

Restrictions cannot be silently removed during role handoffs.

## Evidence Preservation Rules

All collaboration outputs must maintain evidence traceability.

Evidence-bearing collaboration outputs include:

- artifact bundles
- verification reports
- final verdicts
- explicit restriction records
- surface-integration notes

Collaboration reporting must remain reconstructable from repository state.

## Fail-Closed Escalation Rules

When roles encounter:

- unsupported instructions
- ambiguous governance surfaces
- architecture mutation attempts

the system must:

1. fail closed
2. report the boundary explicitly
3. recommend the next safe pipeline or governed path

Improvisational mutation is prohibited.

## Discoverability Rules

Layer 5 must remain discoverable from canonical operator entry surfaces.

At minimum, operators should be able to reach Layer 5 from:

- `README.md`
- `.codex/AGENTS.md`
- `docs/governance/architecture-doctrine.md`

## Explicit Restrictions

This first Layer 5 canon is bounded.

It does not claim:

- runtime multi-agent orchestration
- automatic role delegation
- an autonomous collaboration engine
- universal Codex collaboration enforcement
- independent authority beyond Layers 0 through 4

## Compliance Signals

Layer 5 is being followed correctly when:

- role handoffs remain ordered and bounded
- discoverability updates do not broaden claims
- restrictions survive across planning, integration, verification, and
  reporting
- evidence remains traceable through final verdict output
- unsupported work still fails closed instead of being improvised

## Non-Goals

This canon does not:

- replace the Layer 4 role model
- create runtime sub-agent infrastructure
- prove orchestration or delegation enforcement
- certify collaboration behavior across unimplemented governance domains
