# 087 --- Establish Role-Scoped Codex Sub-Agent Specialization

## 1. Problem Statement

The governance OS currently operates with a unified Codex execution
surface.

While the Layer 3 Codex Rules canon establishes behavioral constraints
for Codex operation, it does not yet define role separation across
different governance responsibilities.

Using a single undifferentiated agent surface introduces several risks:

-   context pollution across unrelated tasks
-   unclear responsibility boundaries
-   inconsistent verification discipline
-   unsafe mutation authority
-   weak governance auditability

Modern agentic development systems mitigate these risks through
role-specialized agents that focus on distinct responsibilities within
the workflow.

Pipeline 087 establishes a canonical role-scoped Codex sub-agent
specialization model for the governance OS.

This specialization structure defines clear agent roles while ensuring
that all roles remain subordinate to:

-   Layer 0 --- Governance Safety Invariants
-   Layer 1 --- Architecture Doctrine
-   Layer 2 --- Template / Scaffold Governance
-   Layer 3 --- Codex Operational Rules

------------------------------------------------------------------------

## 2. Objectives

This pipeline establishes:

1.  canonical Codex sub-agent roles
2.  role authority boundaries
3.  role responsibility scope
4.  cross-role interaction discipline
5.  safe escalation behavior between roles

The specialization model must remain fail-closed and must not grant
agents authority beyond existing governance layers.

------------------------------------------------------------------------

## 3. Canonical Codex Role Model

The governance OS adopts a role-scoped agent architecture.

Each role represents a bounded operational responsibility.

The canonical role set is:

### 3.1 Governance Planner

Responsibilities:

-   interpret instructions
-   classify requests
-   determine governance relevance
-   select next valid pipeline
-   maintain execution sequencing

Planner limitations:

-   cannot declare final verification verdicts
-   cannot mutate canonical doctrine
-   cannot bypass governance routing

------------------------------------------------------------------------

### 3.2 Governance Verifier

Responsibilities:

-   inspect pipeline artifact bundles
-   verify evidence presence
-   validate restriction preservation
-   determine verification verdicts

Verifier limitations:

-   cannot author structural pipelines
-   cannot alter architectural doctrine

------------------------------------------------------------------------

### 3.3 Architecture Auditor

Responsibilities:

-   validate alignment with Layer 1 architecture doctrine
-   detect architecture mutation attempts
-   enforce doctrine preservation

Auditor limitations:

-   cannot change architecture doctrine
-   cannot introduce new architectural surfaces

------------------------------------------------------------------------

### 3.4 Surface Integrator

Responsibilities:

-   maintain canonical documentation surfaces
-   synchronize README, doctrine, registry, and agents files
-   ensure discoverability of governance layers

Integrator limitations:

-   cannot broaden claims
-   cannot alter governance restrictions

------------------------------------------------------------------------

### 3.5 Restriction Keeper (Safety Auditor)

Responsibilities:

-   enforce fail-closed governance behavior
-   detect unsupported or ambiguous requests
-   prevent claim broadening

Restriction Keeper limitations:

-   cannot override established doctrine
-   cannot block valid governed execution without evidence

------------------------------------------------------------------------

### 3.6 Evidence Reporter

Responsibilities:

-   produce bounded summaries
-   generate verification evidence reports
-   record final verdict artifacts

Reporter limitations:

-   cannot reinterpret evidence
-   cannot modify verification results

------------------------------------------------------------------------

## 4. Authority Hierarchy

All Codex sub-agents remain subordinate to the governance layer
hierarchy:

Layer 0 --- Governance Safety Invariants\
Layer 1 --- Architecture Doctrine\
Layer 2 --- Template / Scaffold Governance\
Layer 3 --- Codex Operational Rules\
Layer 4 --- Codex Role Specialization

Role specialization must never override lower-layer restrictions.

------------------------------------------------------------------------

## 5. Cross-Role Interaction Model

Codex sub-agents must follow the governed interaction model.

Planner → selects pipeline\
Integrator → ensures canonical surfaces\
Verifier → validates results\
Reporter → records evidence\
Restriction Keeper → enforces safety\
Auditor → protects architecture doctrine

Agents must not bypass this discipline.

All structural repository mutations must still occur through governed
pipelines.

------------------------------------------------------------------------

## 6. Safe Escalation Discipline

If a role encounters unsupported conditions it must:

1.  fail closed
2.  report the boundary
3.  recommend the next safe pipeline

Roles must not improvise outside governance rules.
